# Outage 2026-05-25: api startup probe budget too small for current EFS size

## TL;DR

During the `v1.4.0-dev.38 -> v1.4.0-dev.39` deploy, every freshly-created `ubtrace-api` (and `ubtrace-worker`) pod failed to bind port 3000 within the chart's 150-second `startupProbe` budget. The bootstrap walk in `UbtSphinxParserService.onModuleInit` scans `/data/input_build` recursively for `ubtrace_project.toml` files; cold NFS metadata walk now takes ~3-4 minutes on prod. Kubelet SIGKILLed every new pod before it could reach Ready. The pre-existing dev.38 pod stayed Up because it had finished the scan once and never had to redo it.

Confirmed (post-recovery) that `ub-backend@dev.39` and `ub-backend@dev.38` are byte-identical (same `main.js` sha256). The only code change in dev.39 is `worker-scheduler.service.ts` (PR #1417 ASPICE fixes). dev.39 itself was not the trigger; the trigger was creating any new api pod against the current EFS state.

Public service went 503 for ~21 minutes when, mid-investigation, the surviving dev.38 pod was deleted to test a "singleton-resource contention" hypothesis. The replacement pod hit the same probe-budget wall.

## Root cause

`apps/ub/backend/src/app/modules/bootstrap/ubt-sphinx-parser.service.ts:onModuleInit` calls `scanAndParseFiles`, which uses the local storage adapter to walk `UBTRACE_INPUT_BUILD_PATH` (set to `/data/input_build` by the chart) recursively for every `ubtrace_project.toml`. NestJS does not call `app.listen()` until every `onModuleInit` resolves, so port 3000 is not bound until the scan returns.

On prod EFS today, `/data/input_build` holds 30+ project versions across `useblocks/ubtrace-spec`, `useblocks/sphinx-needs-demo`, `eclipse-score/score`, `useblocks/prex`, `useblocks/okr`, `useblocks/ubconnect`. Cold NFS metadata walk on a freshly-mounted pod is much slower than warm walk in a long-running pod. Empirically: ~3-4 minutes cold vs near-instant warm.

The chart's `startupProbe: periodSeconds=5, failureThreshold=30` gives the container 150 seconds to bind 3000. Cold scan exceeds that; kubelet SIGKILLs; pod restarts; cold scan again; never converges.

## Contributing factors

- **First investigation pulled toward wrong roots.** Initially diagnosed as a chart-default regression (`runAsGroup`), then as a `RATE_LIMIT_ENABLED=true` Redis-init hang. Both were measurable differences between OLD and NEW pods, both turned out unrelated. Both burned investigation time.
- **Self-inflicted outage to test a hypothesis.** Deleting the surviving OLD pod to test "is the OLD pod holding a singleton resource" was the move that turned investigation into a real public outage. The hypothesis was wrong (the issue was probe budget, not a singleton lock).
- **Helm release pollution.** Five failed `helm upgrade` attempts left rev 25-29 all as "failed" status; recovery had to bypass helm entirely via `kubectl patch` + `kubectl set image`, leaving the live cluster spec drifted from the helm release record.

## What prevented data loss

- RDS snapshots taken at 12:30, both reached `available`. Never needed.
- All ubtrace-managed PVCs annotated with `helm.sh/resource-policy: keep` during the rollback churn; helm could not destroy them on subsequent failed upgrade attempts.

## Workaround in place

- `kubectl patch deployment/ubtrace-api -n ubtrace --type=json -p='[{"op":"replace","path":"/spec/template/spec/containers/0/startupProbe/failureThreshold","value":180}]'`
- `kubectl patch deployment/ubtrace-worker -n ubtrace --type=json -p='[{"op":"replace","path":"/spec/template/spec/containers/0/startupProbe/failureThreshold","value":90}]'`
- Both extend the startup probe budget to 15 minutes. Live cluster spec now diverges from the helm release record at rev 29.

## Real fix tracked

- [useblocks/ubtrace#1418](https://github.com/useblocks/ubtrace/issues/1418): "api startup blocks on EFS scan, exceeds startup probe budget". Recommended: move the bootstrap scan out of `onModuleInit` into a background task, or have the worker persist a project-config snapshot that the api reads as a single file.

## Lessons

- For any chart change that affects bootstrap timing, the cold-pod path needs an explicit test on a prod-shaped dataset before merging; warm pods hide the regression for as long as they stay running.
- Do not delete the only healthy pod of a service to test a hypothesis when there is no instant fallback; the test plan should leave a working replica available.
- Helm release state is fragile when multiple failed upgrades stack; preserve a known-good `deployed` revision before retrying an upgrade.
