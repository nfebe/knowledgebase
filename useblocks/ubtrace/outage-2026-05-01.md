# Outage 2026-05-01: ALBs deleted mid-upgrade

## TL;DR

The prod overlay used during the chart 1.2.0 -> 1.2.1 / app `1.3.0-dev.22` upgrade was reconstructed from `helm get values --all` (rendered output) instead of from the in-repo overlay. Several ingress blocks ended up missing `enabled: true` and `className: alb`. Helm rendered zero ingresses for those hosts, the AWS Load Balancer Controller observed them disappearing and deleted four ALBs. Site dropped on `team.useblocks.com`, `api.`, `admin.`, `auth.` simultaneously.

Recovery took multiple iterations because each fix exposed the next gap (PVC immutability, then StatefulSet immutability, then secret deletion, then healthcheck path, then HTTPS listener missing, then Route53 stale).

No data was lost. RDS, EFS, and the `ubtrace` Secret all survived (after the Secret was restored from a backup taken before the cutover).

## Timeline

| Step | What happened | Lesson |
| --- | --- | --- |
| 1 | First upgrade attempt with reconstructed overlay | PVC update rejected, accessMode immutable. Added `persistence.*` overrides. |
| 2 | Re-run | Elasticsearch StatefulSet rejected, `volumeClaimTemplates.storageClassName` immutable. Added `global.storageClass: gp3` back. |
| 3 | Re-run with `global.existingSecret: ubtrace` cutover | Helm deleted the live `ubtrace` Secret because the previous release manifest still owned it. Restored from backup. Migrate job was failing with `secret "ubtrace" not found`. |
| 4 | Annotated secret `helm.sh/resource-policy: keep`, re-ran | Helm upgrade succeeded, but only `admin` ingress had `enabled: true` -- chart rendered no ingresses for api/frontend/keycloak. AWS LB Controller deleted three ALBs. |
| 5 | Added `enabled: true` and `className: alb` to all four ingresses, re-ran | All four ALBs recreated, but Route53 alias records still pointed at deleted ALB DNS. Updated Route53. |
| 6 | Site loaded but `api.` returned 502 | ALB target group healthcheck path was `/health` (chart default), api on dev.22 returns 404 there. Fixed to `/api/v1/health/readiness`. |
| 7 | Site loaded for `admin.` but `team.` and `api.` and `auth.` failed in the browser | Modern browsers HSTS-upgrade to HTTPS, only `admin` had the HTTPS:443 listener and cert ARN. Added HTTPS listener and cert ARN to all four ingresses, re-ran. |
| 8 | Smoke test: login as `nfebe.fon` and `ubteam` | Both worked. 18 Keycloak users intact. Done. |

Final state: helm release v14, chart `ubtrace-1.2.1`, app images `1.3.0-dev.22`, all 4 ALBs healthy, HTTPS:443 + HTTP:80 on all, target groups healthy, Route53 INSYNC.

## Root cause

A single missing source-of-truth `values-team.yaml` in the repo. The deployment was driven from a `/tmp/values-prod.yaml` that I reconstructed from `helm get values --all`. That output flattens defaults vs overrides and silently dropped chart defaults that mattered (like `ingress.enabled: false`). Every gotcha listed in `chart-quirks.md` was a downstream consequence of working from a derived view instead of the canonical overlay.

## What prevented data loss

- RDS instances are external to the cluster and unaffected by helm operations.
- EFS file system is external and PVC immutability errors **prevented** any destructive PVC change rather than allowing one.
- The `ubtrace` Secret was the only stateful chart-managed resource. It was deleted at step 3, but had been backed up at step 1. Subsequent upgrades cannot delete it because of the `helm.sh/resource-policy: keep` annotation now on the live object.

## What we changed permanently

| Change | Where | Status |
| --- | --- | --- |
| `helm.sh/resource-policy: keep` on `ubtrace` Secret | live cluster | applied |
| `global.existingSecret: ubtrace` adoption | helm release values | applied (revision 14) |
| Cert `aac8197b-...` covers all 4 SANs and is wired into all 4 ingresses | helm release values | applied |
| API ingress healthcheck path `/api/v1/health/readiness` | helm release values | applied |
| Working overlay | `/tmp/values-prod.yaml` | **NOT yet committed**; needs to land as `values-team.yaml` in the EKS PR |

The "NOT yet committed" item is gap #1 in `deploy-gaps.md` and the highest-leverage fix for preventing this from recurring.
